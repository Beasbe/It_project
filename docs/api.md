# Документация API

## Обзор

Этот проект на Next.js поддерживает маршруты API в директории `app/api/`, что позволяет реализовать серверless бэкенд-функциональность. Эти маршруты могут обрабатывать HTTP-запросы (GET, POST и т.д.) и полезны для задач, таких как отправка форм, получение данных или интеграция с внешними сервисами.

В настоящее время директория `app/api/` пуста, что означает отсутствие активных эндпоинтов API в проекте. Эта документация охватывает текущее состояние, как добавить новые маршруты API и потенциальные сценарии использования на основе структуры приложения.

## Текущее состояние API

- **Нет определенных эндпоинтов**: Папка API не содержит файлов маршрутов (например, нет `route.ts` или подобного). В результате серверные API недоступны.
- **Последствия**: Приложение в основном статическое или рендерится на стороне клиента. Любые динамические взаимодействия (например, отправка форм) могут обрабатываться на стороне клиента или через внешние сервисы.

Если API будут добавлены в будущем, этот документ следует обновить с деталями об эндпоинтах, методах, параметрах, ответах и аутентификации.

## Добавление маршрутов API в Next.js

Next.js упрощает создание маршрутов API. Вот руководство для начала:

### 1. Структура директорий
- Создавайте файлы в `app/api/[route]/route.ts` (или `.js` для JavaScript).
- Пример: Для эндпоинта на `/api/contact` создайте `app/api/contact/route.ts`.

### 2. Базовый пример
Создайте простой маршрут API для обработки отправки контактной формы (гипотетический, основанный на компоненте ContactForm):

```/dev/null/example-route.ts#L1-20
import { NextResponse } from 'next/server';

export async function POST(request: Request) {
  try {
    const body = await request.json();
    // Process the form data (e.g., send email, save to DB)
    console.log('Form data:', body);
    
    return NextResponse.json({ message: 'Form submitted successfully' }, { status: 200 });
  } catch (error) {
    return NextResponse.json({ error: 'Failed to submit form' }, { status: 500 });
  }
}
```

- **Методы**: Экспортируйте функции вроде `GET`, `POST`, `PUT` и т.д., соответствующие HTTP-методам.
- **Динамические маршруты**: Используйте `[param]` в именах папок для динамических параметров (например, `app/api/users/[id]/route.ts`).

### 3. Лучшие практики
- **Обработка ошибок**: Всегда обрабатывайте ошибки и возвращайте соответствующие коды статуса.
- **Валидация**: Используйте библиотеки вроде Zod для валидации входных данных.
- **Безопасность**: Защищайте маршруты аутентификацией, если нужно (например, с помощью NextAuth.js).
- **Переменные окружения**: Храните секреты (например, API-ключи) в файлах `.env`.
- **Тестирование**: Тестируйте эндпоинты с помощью инструментов вроде Postman или curl.

### 4. Интеграция с компонентами
- Из клиентских компонентов (например, ContactForm.tsx) используйте `fetch` для вызова API:
  ```/dev/null/example-fetch.ts#L1-10
  const submitForm = async (data) => {
    const response = await fetch('/api/contact', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(data),
    });
    // Handle response
  };
  ```

## Потенциальные сценарии использования API

На основе областей проекта (например, Services, Projects, News, ContactForm) потенциальные API могут включать:

- **Отправка контактной формы**: POST `/api/contact` – Отправка email или хранение запросов.
- **Лента новостей**: GET `/api/news` – Получение последних новостей из внешнего источника или базы данных.
- **Данные проектов**: GET `/api/projects` – Динамическое получение деталей проектов.
- **API партнеров**: POST `/api/partners` – Для административных целей, если добавлена аутентификация.
- **Аналитика**: POST `/api/track` – Логирование взаимодействий пользователей.

Если используются внешние API (например, через зависимости), документируйте их здесь. Например:
- В настоящее время внешние API не интегрированы, но можно добавить библиотеки вроде `axios` для получения данных.

## Соображения безопасности

- **CORS**: Next.js автоматически обрабатывает CORS, но настройте при необходимости.
- **Ограничение скорости**: Реализуйте для публичных эндпоинтов, чтобы предотвратить злоупотребления.
- **API-ключи**: Если требуется, информируйте пользователей (например, "Этот API требует ключ от [сервиса]. Установите его в .env как API_KEY."). Не хардкодьте ключи.
- **Лучшие практики**: Следуйте рекомендациям OWASP; избегайте раскрытия чувствительных данных.

## Тестирование API

- Локально: Запустите `yarn dev` и тестируйте с `curl http://localhost:3000/api/[route]`.
- Инструменты: Postman, Insomnia или Jest для unit-тестов (см. test.md для деталей).

Если API реализованы, расширьте этот раздел с помощью спецификаций Swagger/OpenAPI или детальными таблицами эндпоинтов.
